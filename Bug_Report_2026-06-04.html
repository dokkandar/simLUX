Bug detail

B-01
PLINE arc segments rendered as straight chords once committed
d14f41b

How it surfaced

User drew a polyline with arc segments (PLINE Arc sub-mode), saw the arcs in the live rubber-band preview, then hit Enter — arcs collapsed to straight chord segments the instant the polyline committed.

Root cause

Both draw_dobject_thick (regular + selected highlight) and draw_dobject_dashed (selection outline) had a Polyline arm that flattened bulges to chords. Comment in code even acknowledged it: "Polyline = piecewise lines. Bulges deferred — straight only for now". The kernel data model carried bulge per vertex correctly — the renderer just ignored it.

Fix

New shared helper polyline_tessellated_screen_pts(&Polyline, app, rect) → Vec<Pos2> walks the vertex chain and expands each arc segment by computing (centre, radius, sweep) from the bulge then sampling at on-screen density. Both render paths now route through it — committed polylines render the same arcs the preview shows.

B-02
Hatch ignored nested boundaries and didn't update when source moved
fc762c5

How it surfaced

User selected an outer polyline + 3 inner circles. Old hatch only fired on closed polylines; circles were skipped. Even if hatched, moving the boundary later left a stale fill at the old position — the hatch carried a vertex snapshot, not a reference.

Root cause

v1 Hatch owned boundary: Vec<Vec2> — a copy taken at the moment of apply_hatch. No live link to the source dobjects.

Fix — redesign

Following the LibreCAD model the user pointed at, Hatch now holds boundary_handles: Vec<Handle> — references, not copies. Renderer resolves handles each frame via the new Document::find_by_handle(h) — live link, move/edit propagates automatically.

  • Geom-level transforms (rotated/scaled/mirrored/translated) become no-ops on Hatch — boundaries transform on their own.
  • Multi-loop support via even-odd at render — outer + inner closed polylines become outer fill + holes.

B-03
PLINE drawing showed nothing between clicks ("dots only")
e85b42e

How it surfaced

User clicked points to draw a polyline; nothing appeared between clicks until Enter committed the whole thing. Reported as "pline is full of dots".

Root cause

The draw-tool preview match (self.tool, self.pending.as_slice()) had branches for Line / Circle / Ellipse / EllipseArc but no Polyline branch. The only thing on screen between clicks was the drafting cursor sitting on each captured vertex.

Fix

Added a Polyline preview branch: filled dot per captured vertex, solid hint-color segments between consecutive captured vertices, dashed rubber-band from the last vertex to the cursor. Ortho + grid-snap already worked because polyline's ortho_anchor returns the last pending point.

B-04
Hatch rejected Circle / Ellipse as boundaries
6e058e1

How it surfaced

User's screenshot: outer polyline + 3 inner circles, all selected, ran hatch → "no closed polyline in selection". The circles were geometrically closed; the filter rejected them anyway.

Root cause

v2 hatch filter was match { Polyline(p) if p.closed && p.vertices.len() >= 3 => accept, _ => skip }. Circles and Ellipses fell into the skip arm despite being closed by construction.

Fix

  • apply_hatch filter widened to accept Circle and Ellipse alongside closed Polylines.
  • resolve_hatch_loops tessellates each non-polyline boundary to a 64-vertex closed loop via new helpers tessellate_circle_loop / tessellate_ellipse_loop.

Open chains (line + arc loops) and Splines still skipped — they need boundary-traversal to chain into a loop (queued as Hatch v2b).

B-05
Hatch attributes dialog used OK/Cancel; didn't match the AutoCAD workflow
669151e

How it surfaced

User: "instead of OK / Cancel should be Dobject/s, Pick point". The dialog's OK button left the user mid-air — how does the boundary get selected after OK? AutoCAD-style workflow puts the boundary-acquisition mode on the dialog.

Root cause

Design mistake on my part — OK/Cancel maps to "commit/discard a pending value", which doesn't fit the hatch flow where the user still needs to acquire the boundary AFTER pattern selection.

Fix

  • ☐ Dobject/s → begin ForSelect session for boundary dobjects; Enter commits with the dialog's pattern.
  • ◉ Pick Point → arm canvas; next pointer-mode click runs find_smallest_containing_closed on every closed dobject in the doc (PIP test + bbox-area tiebreak), hatches the winner.
  • X (window close) → dismiss, no commit.

New helper point_in_polygon(p, iter) — standard half-open Y-test PIP.

B-06
No real-time visibility into the hatch state machine
135d635

How it surfaced

User: "something is wrong, we need to real-time debug — add a debug menu, record all the display dobjects and screen events". Without a per-event log, hatch failures came back as "nothing happened".

Fix

New Hatch Debug Log window mirroring the existing Trim Debug pattern. Toolbar toggle (hatch dbg ▸). Records every state transition: command parse / dialog open / dialog button / pick-point click / accept-skip per dobject / apply_hatch params / hatch pushed / X close. Live status badge (idle / dialog open / pick-point armed / awaiting boundary). Pending-pattern always-visible readout so you can SEE what the next apply_hatch will commit. Buttons: 📋 Copy Log, 🗑 Clear, 📸 Dump Hatch State (snapshot every Hatch dobject + resolved-loop diagnostics).

hatch_dbg(msg) is a no-op when the window is closed — wide instrumentation doesn't cost memory in normal use.

B-08
GPU render mode silently dropped every Hatch dobject
58c1281

How it surfaced

Found by debug-log analysis after B-07: user's log showed 3 hatches successfully committed (apply_hatch all OK), but they didn't see them. CPU branch was short-circuiting hatches to render_hatch_fill; GPU branch's _ => arm sent them to the free draw_dobject, which is stubbed for Hatch.

Fix

GPU's non-circle arm now checks if let Geom::Hatch(h) and dispatches to render_hatch_fill (the only path with Document access for handle resolution) before falling through to draw_dobject.

Plus diagnostics

  • apply_hatch now auto-calls dump_hatch_state at its tail (no need to press the button).
  • dump_hatch_state enriched with per-loop bbox + per-family line-count estimate. ⚠ ZERO LINES warning when spacing > boundary diagonal — catches "pattern looks empty" caused by too-coarse default scale.
B-09
GPU mode painted every dobject in one of three hardcoded colors
77a0759

How it surfaced

User: "in GPU the dobjects are not getting colour as single colour, in CPU all has got the colour, is it architecture problem?"

Root cause

GPU branch hardcoded:

let sel_col  = 0xA0A5AFFFu32;  // gray  — selected
let snap_col = 0x78F0FFFFu32;  // cyan  — snap source
let def_col  = 0xAAC8E6FFu32;  // light blue — everything else

Per-dobject style.color (ByLayer / ByBlock / ACI / TrueColor) was completely ignored. CPU resolved correctly via resolve_color(e.style.color, e.style.layer, &doc.layers, &doc.truecolors); GPU just never called it.

Fix

GPU's color resolution now matches CPU. Selection (yellow) and snap-source (cyan) overrides applied first; everything else routes through resolve_color then packs to the shader's expected byte order (R<<24)|(G<<16)|(B<<8)|A, matching the vertex-shader unpack at gpu.rs:66. Applies to both Circle (instance buffer u32) and non-circle (egui Color32).

No shader / kernel change — shader was always ready to take arbitrary RGBA per instance; CPU side just wasn't feeding real colors.

B-10
CPU and GPU hatch render diverged: GPU showed hatches, CPU didn't
9f1571c

How it surfaced

User's screenshots: GPU rendered the hatch correctly inside a circle; CPU rendered only the circle outline. Two render paths producing different results on the same doc — clear architectural mismatch.

Root cause — three layered bugs, all from Hatch::bbox() = (0,0)

  1. Spatial index — UniformGrid::build bucketed Hatch in only the origin cell. query_bbox(viewport) returned it only when the camera included the origin. Pan away → never in candidate_iter.
  2. Viewport cull at the top of the CPU per-loop checked Hatch's bbox against (v_min, v_max). With bbox (0,0) the cull dropped Hatch unless origin was in view.
  3. State-based branches inside the CPU loop (selection-dashed, trim-pulse) routed Hatch into draw_dobject_dashed / draw_dobject_thick — both stub Hatch as no-op. Hatch in selection basket → drawn as nothing.

GPU branch happened to dodge (2) and (3) because its loop has no state-based dispatch — every non-Circle reached the renderer. That's why GPU appeared to "just work" while CPU didn't, even though the underlying bbox bug was in both.

Fix

  • Kernel — new Geom::is_view_independent_bbox() -> bool. Returns true for Hatch (and future Region/BlockRef/Xref). Default false.
  • Spatial index — UniformGrid tracks view_independent: Vec<u32> separately. Build skips them when computing the world bbox + cell-size and when bucketing into cells. query_bbox and query_near always append the full list — camera position doesn't matter.
  • App — top-of-CPU-loop Hatch short-circuit before viewport cull, micro-cull, and any state-based branch. Dispatches straight to render_hatch_fill with the correctly-resolved color.
B-11
Hatch pattern looks empty at default scale on small shapes
58c1281

How it surfaced

Inferred from B-08 debug analysis. ANSI31's catalog spacing is 3.175 world units (the AutoCAD default = 1/8 inch). On a polygon ~2 units wide, 0 or 1 pattern lines cross the boundary — reads as "no hatch".

Fix — diagnostic, not behavior

dump_hatch_state now emits a per-family line-count estimate and flags zero-line families:

hatch #7 pattern=[ANSI31 scale=1.000 angle=0.00]
  boundary_handles=[#5] resolved_loops=[5v bbox=2.10x3.40]
  estimated families: [45(1 lines @ spacing 3.175)]
  ⚠ ZERO LINES — raise scale or pick a finer pattern

User-actionable: hatch ANSI31 10 scales 10× and pattern becomes visible. Long-term, a "Fit Pattern to Boundary" auto-scale checkbox in the dialog would remove the friction; queued for the hatch UX polish slice.

Architectural lessons

L-1
By-reference entities need first-class kernel support

Hatch is the first dobject whose bbox / distance / contains depends on data the kernel can't see (boundary dobjects in the Document). Returning a placeholder (0,0) exposed three independent failure modes (B-10).

Same trap waits for every future by-reference entity:

  • Region (Hatch v2c) — boundary refs
  • BlockRef — block definition ref
  • Xref — external drawing ref
  • Group — member refs
  • Dimension — sometimes refs the dimensioned dobject

Today's mitigation: the is_view_independent_bbox() trait-method pattern + spatial-index global list. Works for any number of new entity types — each one just answers true and the spatial path handles it correctly.

Permanent fix queued under Handle-first Document API refactor: kernel methods that need Document access should take it explicitly. fn bbox(&self, doc: &Document) -> (Vec2, Vec2) instead of a free fn bbox(&self). Touches every call site, but eliminates the placeholder-bbox class of bugs.

L-2
State-based render dispatch is fragile against new Geom types

The CPU render loop has multiple state-based early-exit branches: layer-visible, viewport-cull, micro-cull, snap-source-cyan, selection-dashed, trim-cutter-pulse, extend-boundary-pulse, then finally draw_dobject. Each branch calls a render function with its own per-Geom match arms. Add a new Geom variant and you have to remember to handle it in EVERY branch's render function.

Today's Hatch slipped because:

  • Stubbed in draw_dobject_thick when introduced
  • Stubbed in draw_dobject_dashed when introduced
  • Worked in the "normal" branch only, because I added the short-circuit there

4× the work per entity type, and easy to forget — which is exactly how this happened.

Permanent fix queued: a per-Geom render trait method (e.g. trait Renderable { fn render_into(&self, ctx: &RenderCtx); }) so dispatch lives WITH the geometry, not in match-arm pyramids in the render loop. The state (selected / cutter / snap source / etc.) becomes RenderCtx fields. Each new entity then only has ONE place to think about.

L-3
GPU and CPU render paths drift unless they share dispatch

B-09 (GPU ignored per-dobject color) and B-10 (GPU/CPU diverged on Hatch) are the same shape of mistake: GPU branch was written as a simpler version of CPU, with a different short-circuit structure. Each fix to one had to be remembered in the other; bugs accumulated when memory failed.

Today's mitigation: B-09 and B-10 both audit-and-mirror — GPU now uses the same resolve_color + the same Hatch dispatch as CPU.

Permanent fix (same as L-2): a shared render_into(ctx) per Geom, with the backend selection (egui Painter vs glow instance buffer) inside ctx. Then there is exactly one code path; the only "split" is whether each render call falls into the painter or the instance buffer at the lowest level — not whether the dispatch even happens.
