Let's break this down into a comprehensive research document and a development roadmap.

### Phase 1: Comprehensive Research & Foundation

Before writing a single line of Rust, we must define the mathematical and physical models we're implementing. A "good enough" simulation for an electrician is very different from a physically accurate one for a lighting designer. We'll aim for a physically-based, radiosity-capable model suitable for professional use.

#### 1.1 Core Photometric & Physical Models

This is the heart of the application. We cannot cut corners here.

- **IES File Parsing (LM-63 Standard):**
    - **Challenge:** IES files are deceptively complex. They contain metadata, TILT data, and multiple angular measurement formats (Type A, B, C photometry).
    - **Solution:** We must build a robust parser for the IES LM-63-2002 (and later) standard.
    - **Key Data Points to Extract:**
        - Number of vertical angles and their array.
        - Number of horizontal angles and their array.
        - The 2D array of candela values (`candela[vert_angle][horiz_angle]`).
        - Multiplying factor for candela values.
        - Luminaire dimensions (length, width, height) for luminous opening.
    - **Interpolation:** We'll rarely query an exact angle from the file. Bicubic or at least bilinear interpolation on the candela table is essential for smooth results. A simple nearest-neighbor approach will produce a visibly jagged, unprofessional light distribution.

- **Light Source Model:**
    - **Point Source Approximation:** The simplest. A luminaire is a single point in 3D space `(x, y, z)`. Valid for far-field calculations where distance > 5x the maximum luminaire dimension.
    - **Luminous Area Model (Recommended):** A luminaire is a rectangular polygon. This is crucial for accurate near-field photometry (e.g., a light close to a wall or the floor). We'd approximate this by discretizing the area into a grid of sub-point sources, each emitting a fraction of the total intensity.

- **Surface & Material Model:**
    - A `Surface` is a planar polygon (wall, floor, ceiling).
    - **Core Property: Reflectance.** A simple RGB or scalar value (0-1) representing the percentage of light reflected.
    - **Reflection Model: Lambertian (Diffuse Only).** This is the standard assumption for interior lighting calculations (radiosity method). Light hitting a point on a wall is reflected equally in all directions. Specular reflections (like on a mirror) are an advanced feature and computationally far more expensive.

#### 1.2 The Calculation Engine: Radiosity is King

Direct-only calculation (luminaire -> floor) is trivial but useless for any real-world scenario with walls. We need to account for light bouncing off surfaces. The **Radiosity method** is the classic, proven algorithm for this.

**The Radiosity Equation (Simplified):**
$$B_i = E_i + \rho_i \sum_{j=1}^{N} B_j F_{ij}$$

Where:
- $B_i$ = Radiosity (total light leaving) of surface $i$.
- $E_i$ = Direct emitted light from surface $i$ (non-zero for our luminous area sources).
- $\rho_i$ = Reflectance of surface $i$.
- $F_{ij}$ = **Form Factor** between surface $i$ and surface $j$.

**The Core Problem: The Form Factor ($F_{ij}$)**
The form factor dictates what fraction of light leaving a perfectly diffuse surface $j$ arrives at surface $i$. Calculating this for every pair of polygons in the scene is the computational bottleneck. A naive approach is $O(N^2)$.

- **The Hemicube Method (Standard):**
    - To compute $F_{i, all\_others}$, you place a virtual camera at the center of polygon $i$, pointing along its normal.
    - You render the scene onto the five faces of a hemicube (a half-cube).
    - Each pixel on the hemicube represents a small solid angle. The polygons visible in the pixel are determined using a **z-buffer**.
    - We sum the contribution of each pixel, weighted by its pre-calculated "delta form factor." This is a classic computer graphics problem.

- **A Pragmatic, Modern Alternative: Ray Tracing.**
    - Ray tracing can be seen as a Monte Carlo solution to the rendering equation, which includes the radiosity equation.
    - **Algorithm:**
        1.  For a surface point to be calculated (e.g., a grid point on the floor), shoot $N$ rays in a cosine-weighted hemisphere.
        2.  For each ray, find the first surface it hits.
        3.  If it hits a luminaire, add the luminaire's direct luminance.
        4.  If it hits another surface (wall), recursively compute the light leaving that point, or just add its calculated radiosity from the previous frame/iteration.
    - **Why this might be better for us:** It's conceptually simpler, automatically handles any geometry without hemicube discretization artifacts, and scales beautifully with modern multi-core CPUs. Rust's fearless concurrency makes a parallel ray tracer an excellent fit.

**Recommendation:** Implement a **Progressive Radiosity via Ray Tracing**. We store the "accumulated unshot radiosity" for each surface. In each iteration, we process the surface with the most unshot light, shoot rays from its samples, and "shoot" that light onto other surfaces. The floor lux grid is just a set of sensor points that participate in this process.

#### 1.3 Computational Geometry Engine

You cannot escape this. This is a massive part of the codebase.

- **Robust 2D Boolean Operations:** The user will draw a rectangle, then another, and "trim" an opening for a door. We need a stable library to perform union, intersection, and difference on 2D polygons. This forms the final shape of the walls.
- **3D Extrusion:** A wall is defined by its 2D footprint (a closed polygon with wall-thickness) and an extrusion height.
- **Triangulation:** GPUs and many geometric algorithms eat triangles. Every surface (wall polygon, floor, ceiling) must be broken down into a mesh of triangles for the rendering/ray-tracing step.
- **Point-in-Polygon Test:** Essential for converting the user's click on a DXF layout into a selection of a room or wall. A robust "ray casting algorithm" is needed.
- **DXF Parsing:** A DXF file is a database of graphical entities (LWPOLYLINE, LINE, ARC, CIRCLE, INSERT). We need a parser that can read these, convert them into our internal geometric representations (polygons), and handle layers. The most critical entity is `LWPOLYLINE` which will define walls.

### Phase 2: Rust/Tauri Technology Stack & Architecture

This maps our research onto a concrete software design.

#### 2.1 Crate Selection (Rust Libraries)

This is the "secret sauce" choice.

- **GUI (Tauri Frontend):**
    - We'll use a modern web framework like **Svelte, SolidJS, or React** with a `<canvas>` for the 2D/3D viewport. The GUI handles user input, not heavy computation.
    - **Three.js** is the obvious choice for the `<canvas>`. It handles DXF visualization, wall drawing tools, and can even display a raw preview of the ray-traced result.

- **Rust Backend (The Engine):**
    - **DXF Parsing:** `dxf-rs` (check its maturity, may need to write custom extensions or use a simpler parser as a base).
    - **Geometry & Meshing:**
        - `geo`: Powerful 2D computational geometry library. Polygons, boolean ops, point-in-polygon tests. An absolute must-have.
        - `parry2d`/`parry3d`: For advanced collision detection, ray casting against meshes.
        - `earcutr`: Fast, robust polygon triangulation to turn our complex wall shapes into meshes for the ray tracer.
    - **Math:**
        - `nalgebra` or `glam`: For vectors (`Vec3`), matrices (`Mat4`), quaternions. `glam` is simpler and faster for game-like workloads; `nalgebra` is more feature-complete.
    - **Ray Tracing Engine (The Core):** We will write this from scratch. It's not a library we just drop in; it is the defining feature. We'll use `rayon` for simple, work-stealing parallelism across the lux grid calculation. For a single point's irradiance calculation, we'll shoot hundreds of rays.
    - **IES Parsing:** We'll write a custom parser using a crate like `nom` or `pest`. IES files have a very strict, old-school column-dependent text format, making a parser combinator ideal.
    - **State Management & Interop:**
        - We'll define all shared structures in Rust: `Project`, `Room`, `Wall`, `LuminaireInstance`, `IesProfile`, `LuxGrid`, `RayTracingSettings`.
        - `serde` will serialize these to JSON to send to the frontend for visualization and to save/load `.lux` project files.
        - Tauri `command`s will be the API: `load_dxf()`, `add_wall()`, `import_ies()`, `calculate_lux()`.

#### 2.2 Architecture Diagram (Conceptual)

```text
[Tauri Frontend (Svelte + Three.js)]
  |  User draws walls, places lights on a 2D/3D canvas.
  |  Sends events via Tauri `invoke`.
  v
[Tauri Command Layer (Rust)]
  |  Deserializes commands. Owns the `AppState`.
  |
  v
[Core Engine (Rust)]
  |
  |-- Project Manager: Owns the project tree (Rooms, Luminaires, Materials).
  |-- IES Parser: Async task to parse and validate .ies files.
  |-- Geometry Module:
  |   |-- DXF Parser: Converts DXF -> internal Room/Wall geometry.
  |   |-- Wall Builder: 2D boolean ops (union, subtract) for walls.
  |   |-- Mesher: Extrudes 2D walls into 3D, triangulates for the engine.
  |
  |-- Calculation Module:
      |-- Scene Builder: Gathers all meshes, materials, and light sources.
      |-- Ray Tracer: The progressive radiosity engine.
      |   |-- Direct Lighting Pass: For each floor grid point.
      |   |-- Indirect Lighting Pass: Recursive ray bounces.
      |   |-- Accumulator: Stores and averages path results.
      |-- Lux Grid Output: A 2D array of lux values.
```

### Phase 3: The Development Roadmap

This is a phased, iterative plan. Each phase delivers a working application.

#### Phase 3.1: MVP - Direct-Only Calculation on Simple Floor

- **Goal:** Load a DXF, manually define a single floor rectangle, place a single IES luminaire, and get a direct-only lux grid.
- **Frontend:**
    - `file` input for .dxf.
    - A `<canvas>` showing parsed DXF lines.
    - A tool to click-drag a "Calculation Plane."
    - A tool to drag an imported light onto the plan (sets its XY, user types Z height).
    - A "Calculate" button.
    - Display the resulting lux grid as a heatmap on the canvas.
- **Backend (Rust):**
    - **State:** `AppState` containing a list of 2D lines from DXF, one `CalculationPlane` (a quad in 3D), one `LuminaireInstance` (position + IES data).
    - **DXF Parser:** Can read `LWPOLYLINE` from a file and return a `Vec<Line>`.
    - **IES Parser:** Can parse Type C photometry and perform linear interpolation on the candela table.
    - **Direct Calculation Engine:**
        - Discretize the `CalculationPlane` into an `M x N` grid of points.
        - For each point `P`:
            - Vector `L` = `luminaire.position - P`.
            - Perform shadow test: Shoot a single ray from `P` to luminaire's center. If it hits a "wall" (for now, just a few hardcoded infinite planes), result is 0.
            - If not in shadow, compute $E = \frac{I(\theta, \psi) \cdot \cos(\epsilon)}{d^2}$, where $\epsilon$ is the angle of incidence with the floor normal.
        - Return the `Vec<f64>` lux grid.

#### Phase 3.2: Wall Drawing & Basic 3D Scene

- **Goal:** Let the user trace the DXF lines to create 3D rooms with wall-thickness.
- **Frontend:**
    - A dedicated "Draw Wall" tool.
    - Snapping to DXF endpoints and midpoints.
    - A "Room" panel showing a list of rooms.
- **Backend:**
    - **Geometry Engine:** A `Room` struct is a closed loop of `Wall` segments. A `Wall` is a 2D polygon (with thickness) extruded to a user-defined height.
    - **Triangulation:** Each wall's 3D surfaces are triangulated into a `Mesh` (`Vec<Triangle>`).
    - **Enhanced Ray Casting:** The shadow test from Phase 3.1 now uses `rayon` and does a brute-force (or BVH-accelerated) `ray vs. triangle` intersection test against all walls and the floor.

#### Phase 3.3: Radiosity & Indirect Lighting

- **Goal:** Walls illuminate each other, and the floor. The calculation becomes physically accurate for interior spaces.
- **Backend:**
    - **Material System:** Each `Mesh` gets `Reflectance`.
    - **Progressive Radiosity Engine:**
        - **Initialization:** The "unshot radiosity" of each luminaire's polygon is its total lumen output. All other surfaces have zero.
        - **Iteration:** Find the surface with the highest unshot radiosity. Discretize it into sample points. Shoot thousands of rays into its hemisphere. If a ray hits floor grid points, accumulate the illuminance. If it hits another wall, add to that wall's "unshot" buffer.
        - **Convergence:** Run until a set number of bounces or the remaining unshot energy is below a threshold.
        - The floor lux grid is now physically correct.

#### Phase 3.4: Professional Features & Polish

- **Multi-Luminaire:** Support importing and placing hundreds of lights. The BVH (Bounding Volume Hierarchy) acceleration structure for ray casts becomes mandatory for performance.
- **Luminaires as Luminous Areas:** Instead of point sources, model a 2x4 troffer as a 4x4 grid of sub-sources.
- **Colored Light & Surfaces:** Change radiosity from scalar to `(R, G, B)`. The result is not just lux but full-color irradiance.
- **Calculation Points:** Allow the user to place discrete sensor points for spot readings.
- **UGR / Glare Calculations:** A major advanced feature requiring very specific view-position ray tracing.
- **Report Generation:** Export calculated lux grids to Excel/CSV or a PDF report.

This roadmap provides a solid, professional-grade path. The core challenge is undoubtedly the robust, parallel ray-tracing engine for radiosity. Writing it in Rust with Tauri for the UI is an excellent choice, providing the perfect balance of raw performance and a modern development experience. Start with Phase 3.1, get a pixel to light up on the floor, and you'll have proven the entire pipeline.
